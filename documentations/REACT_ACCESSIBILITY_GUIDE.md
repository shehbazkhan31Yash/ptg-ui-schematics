# React Accessibility Tools Integration Guide

## Overview

This guide explains the accessibility tools integration for React applications generated by the PTG UI Schematics. When you enable accessibility tools during project generation, you get a comprehensive suite of tools and components to build accessible applications.

## 🎯 What's Included

### 1. **Axe-core** (v4.8.3)
Automated accessibility testing engine that runs in your browser and catches WCAG violations.

### 2. **@axe-core/react** (v4.8.3)
React integration for axe-core that automatically tests your components during development.

### 3. **eslint-plugin-jsx-a11y** (v6.8.0)
ESLint plugin that catches accessibility issues in JSX at compile time.

### 4. **Accessible Components Library**
Pre-built, accessible React components:
- SkipLink
- LiveRegion
- VisuallyHidden
- FocusTrap
- A11yAnnouncer

### 5. **Utility Functions**
Helper functions for common accessibility patterns:
- Screen reader announcements
- Focus management
- ID generation
- Motion preference detection

## 📦 Installation

When you create a React project with accessibility enabled:

```bash
# During project creation
? Enable accessibility tools? (axe-core, aria-lint, accessible templates) Yes
```

This automatically adds to your package.json:

```json
{
  "dependencies": {
    "axe-core": "^4.8.3",
    "@axe-core/react": "^4.8.3",
    "eslint-plugin-jsx-a11y": "^6.8.0"
  }
}
```

## 🚀 Quick Start

### 1. Initialize Axe in Development

In your main app file (e.g., `src/main.tsx` or `src/index.tsx`):

```typescript
import React from 'react';
import ReactDOM from 'react-dom/client';
import App from './App';
import { initializeAxe } from './app/accessibility/utils/axe-helper';

// Initialize axe-core in development mode only
if (process.env.NODE_ENV !== 'production') {
  initializeAxe(React, ReactDOM, 1000);
}

ReactDOM.createRoot(document.getElementById('root')!).render(
  <React.StrictMode>
    <App />
  </React.StrictMode>
);
```

### 2. Use Accessible Components

```typescript
import { SkipLink, LiveRegion, FocusTrap } from './app/accessibility/components/A11yComponents';

function App() {
  return (
    <>
      <SkipLink targetId="main-content" />
      <LiveRegion id="announcements" />
      
      <header>
        <nav>Navigation</nav>
      </header>
      
      <main id="main-content">
        <h1>Welcome</h1>
        <FocusTrap active={isModalOpen}>
          <div className="modal">
            <h2>Modal Title</h2>
            <button onClick={closeModal}>Close</button>
          </div>
        </FocusTrap>
      </main>
    </>
  );
}
```

### 3. Use Utility Functions

```typescript
import { 
  announceToScreenReader, 
  trapFocus, 
  prefersReducedMotion 
} from './app/accessibility/utils/a11y-utils';

// Announce updates to screen readers
const handleSave = () => {
  saveData();
  announceToScreenReader('Data saved successfully', 'polite');
};

// Check for reduced motion preference
const shouldAnimate = !prefersReducedMotion();

// Trap focus within a modal
useEffect(() => {
  if (isModalOpen && modalRef.current) {
    trapFocus(modalRef.current);
  }
}, [isModalOpen]);
```

## 🧪 Testing with Axe-core

### Automatic Testing in Development

Axe-core runs automatically in development mode and reports violations to the console:

```
Accessibility violations found:
  1. button-name: Buttons must have discernible text
     Impact: critical
     Element: <button></button>
     
  2. color-contrast: Elements must have sufficient color contrast
     Impact: serious
     Element: <p style="color: #777;">...</p>
```

### Manual Testing

```typescript
import { runA11yCheck } from './app/accessibility/utils/axe-helper';

// Test a specific component
const testComponent = async () => {
  const element = document.getElementById('my-component');
  const results = await runA11yCheck(element);
  
  if (results?.violations.length) {
    console.error('Violations:', results.violations);
  }
};
```

### Using Custom Rules

```typescript
import { initializeAxe, axeConfigs } from './app/accessibility/utils/axe-helper';

// Test against WCAG 2.1 AA standards
initializeAxe(React, ReactDOM, 1000, axeConfigs.wcag21aa);

// Or create custom config
const customConfig = {
  rules: {
    'color-contrast': { enabled: true },
    'heading-order': { enabled: true },
    'label': { enabled: true },
    'duplicate-id': { enabled: false } // Disable specific rules
  }
};

initializeAxe(React, ReactDOM, 1000, customConfig);
```

## 📝 ESLint Integration

The accessibility tools automatically configure ESLint with jsx-a11y rules based on your linter choice:

### For Airbnb Config
jsx-a11y is included by default with recommended rules.

### For Custom/Standard Config
jsx-a11y is added when accessibility is enabled.

### Common Rules
- `jsx-a11y/alt-text`: Images must have alt text
- `jsx-a11y/anchor-is-valid`: Anchors must be valid
- `jsx-a11y/aria-role`: ARIA roles must be valid
- `jsx-a11y/click-events-have-key-events`: Click handlers need keyboard equivalents
- `jsx-a11y/label-has-associated-control`: Labels must be associated with controls

### Fixing ESLint Errors

```typescript
// ❌ Bad - Missing alt text
<img src="logo.png" />

// ✅ Good
<img src="logo.png" alt="Company Logo" />

// ❌ Bad - Div with click handler (not keyboard accessible)
<div onClick={handleClick}>Click me</div>

// ✅ Good - Button is keyboard accessible
<button onClick={handleClick}>Click me</button>

// ❌ Bad - Link without href
<a onClick={handleClick}>Click</a>

// ✅ Good - Button for non-navigation actions
<button onClick={handleClick}>Click</button>
```

## 🎨 Component Examples

### Skip Link
Allows keyboard users to skip navigation:

```typescript
<SkipLink targetId="main-content" text="Skip to main content" />
```

### Live Region
For dynamic content announcements:

```typescript
const [status, setStatus] = useState('');

<LiveRegion id="status-messages" ariaLive="polite">
  {status}
</LiveRegion>

// Update status
setStatus('Item added to cart');
```

### Visually Hidden
Hide content visually but keep it for screen readers:

```typescript
<button>
  <VisuallyHidden>Submit form</VisuallyHidden>
  <IconSubmit />
</button>
```

### Focus Trap
Trap focus within modals/dialogs:

```typescript
<FocusTrap active={isModalOpen}>
  <dialog open={isModalOpen}>
    <h2>Confirmation</h2>
    <p>Are you sure?</p>
    <button onClick={confirm}>Yes</button>
    <button onClick={cancel}>No</button>
  </dialog>
</FocusTrap>
```

### A11y Announcer
Global announcer using React portals:

```typescript
import { A11yAnnouncer } from './app/accessibility/components/A11yComponents';

function App() {
  return (
    <>
      <A11yAnnouncer />
      {/* Your app content */}
    </>
  );
}

// Use the global announcer
document.getElementById('a11y-announcer')!.textContent = 'Form submitted';
```

## ✅ Best Practices

### 1. Semantic HTML
Use semantic elements instead of divs:

```typescript
// ❌ Bad
<div className="header">
  <div className="nav">...</div>
</div>

// ✅ Good
<header>
  <nav>...</nav>
</header>
```

### 2. ARIA Labels
Add descriptive labels:

```typescript
<button aria-label="Close modal" onClick={close}>
  <XIcon />
</button>
```

### 3. Form Labels
Always associate labels with inputs:

```typescript
<label htmlFor="email">Email</label>
<input id="email" type="email" name="email" />
```

### 4. Keyboard Navigation
Ensure all interactive elements are keyboard accessible:

```typescript
<div 
  role="button" 
  tabIndex={0}
  onClick={handleClick}
  onKeyDown={(e) => e.key === 'Enter' && handleClick()}
>
  Click me
</div>
```

### 5. Color Contrast
Ensure sufficient contrast (4.5:1 for normal text, 3:1 for large text):

```css
/* ❌ Bad - Low contrast */
.text {
  color: #777;
  background: #fff;
}

/* ✅ Good - High contrast */
.text {
  color: #333;
  background: #fff;
}
```

## 🐛 Common Issues and Fixes

### Issue: Axe not reporting violations
**Solution**: Make sure you're in development mode and axe is initialized before React renders.

### Issue: ESLint jsx-a11y rules not working
**Solution**: Ensure `eslint-plugin-jsx-a11y` is in package.json and eslint.config.js includes the plugin.

### Issue: Focus trap not working
**Solution**: Ensure the FocusTrap component wraps focusable elements and `active` prop is properly toggled.

### Issue: Screen reader not announcing updates
**Solution**: Use LiveRegion with appropriate `ariaLive` level ('polite' or 'assertive').

## 🎓 Testing Checklist

- [ ] All images have alt text
- [ ] All form inputs have associated labels
- [ ] All interactive elements are keyboard accessible
- [ ] Color contrast meets WCAG AA standards (4.5:1)
- [ ] Focus indicators are visible
- [ ] Skip navigation link is present
- [ ] ARIA landmarks are used appropriately
- [ ] Dynamic content updates are announced
- [ ] Modals trap focus correctly
- [ ] No accessibility violations in axe-core report
- [ ] ESLint shows no jsx-a11y errors

## 💡 Tips

1. **Run axe-core regularly**: Keep the console open during development to catch violations early.
2. **Use semantic HTML**: It's accessible by default.
3. **Test with keyboard only**: Try navigating your app without a mouse.
4. **Test with screen reader**: Use NVDA (Windows), JAWS (Windows), or VoiceOver (Mac).
5. **Check color contrast**: Use browser DevTools or online contrast checkers.
6. **Respect user preferences**: Honor `prefers-reduced-motion` and `prefers-color-scheme`.
